# v1.0 vs v1.1 backlog split

Date: 2026-04-20
Purpose: separate launch-critical work from immediate post-launch and longer-horizon improvements, based only on current repo/docs state.

## Boundary summary

- **v1.0 (must finish before launch):** launch evidence capture, runbook/doc correctness for current implemented flows, and removing ambiguity around auth/security boundaries already marked as TODO in schema/policies.
- **v1.1 (first week + candidate set):** automation and hardening around already-shipped paths (claims, withdrawals, realtime, external sync, reconciliation), plus wiring currently stubbed schedulers/workers that are not launch blockers.
- **Longer-term defer:** clearly stubbed subsystems and expansion work not required for launch safety.

## A) Must finish before launch (v1.0)

### auth/security
- **Finalize and document production auth posture for current API header model** (what is accepted in prod vs dev for `x-user-id`, `x-admin-token`, and dev override gating). This is implemented but easy to misconfigure without an explicit launch note and env guard validation. (`apps/web/src/app/api/auth.ts`, `infra/docs/runbooks/launch-checklist.md`).
- **Close the known RLS policy TODO ambiguity for launch signoff** by explicitly documenting the accepted temporary policy scope and owner/date for follow-up. The migration itself explicitly flags unfinished policy tightening. (`supabase/migrations/0012_rls_policies.sql`).

### docs/ops
- **Run and attach DB-backed smoke evidence in CI/staging as launch signoff artifact** (`latest.log` + `latest.json`) per launch-readiness blocker. (`infra/docs/launch-readiness.md`, `infra/docs/release-candidate-freeze-plan.md`).
- **Correct claims runbook drift to match current API routes** (`GET /claims`, `GET /claims/:marketId/state`, `POST /claims/:marketId`) so operators are not told a capability is missing when it exists. (`services/api/src/server.ts`, `infra/docs/runbooks/market-resolution-claims.md`).

### ledger/reconciliation
- **Require one clean reconciliation run in release evidence pack** (already specified, needs execution discipline) before go-live approval. (`infra/docs/release-candidate-freeze-plan.md`, `infra/docs/runbooks/reconciliation-worker.md`).

## Recommended post-launch sequence (first 2 weeks)

### Week 1 (stabilization + automation)
1. Stand up CI `smoke:db` job with artifact retention and make it visible in release dashboard.
2. Wire cron -> external-sync worker trigger and add basic run success/failure metrics.
3. Update withdrawal operator workflow to reduce manual mistakes (queue visibility + standardized action checklist).
4. Add WS gap alert thresholds and explicit on-call runbook links in incident docs.

### Week 2 (hardening + debt retirement)
1. Auth/security cleanup pass: middleware/route protection policy alignment + documented prod header posture.
2. External/internal integration hardening: signed internal webhook + claim notification/evidence stubs moved to real implementations.
3. Decide fate of matching-worker engine TODO lane (implement vs remove dead scaffolding) and begin settlement worker wiring if retained.
4. Kick off scoped RLS hardening design doc for write policies/role boundaries, with migration plan for next release train.
